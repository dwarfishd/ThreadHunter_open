# Permissions

## Доступ к Telegram

Пользователь сам управляет своими API-credentials. Продукт не хранит пароли, только токены локально.

## Модель доступа

### Локальные API-ключи
- Пользователь создаёт приложение на my.telegram.org
- Получает `api_id` и `api_hash`
- Вводит их в конфигурацию продукта (CLI config / файл)
- Ключи хранятся локально, не отправляются никуда кроме Telegram API

### Сессионная авторизация
- При первом запуске продукт запрашивает phone number → code → создаёт session
- Session файл хранится локально рядом с базой данных
- При истечении сессии — продукт просит пользователя повторно авторизоваться

### Что продукт может
- Читать сообщения из публичных и приватных каналов, где пользователь состоит
- Скачивать текст, фото, метаданные постов
- **Не может**: отправлять сообщения, удалять посты, менять настройки каналов

### Что продукт НЕ делает
- Не хранит пароли в открытом виде
- Не отправляет credentials на сторонние серверы
- Не запрашивает доступ к личным чатам пользователя
- Не публикует контент от имени пользователя

## Безопасность

- Session файл — чувствительные данные, пользователь сам защищает свою машину
- Нет облачного бэкапа сессии
- При компрометации session — пользователь может отозвать его через my.telegram.org
