# Operational Risks — Post-Mortem

## O1. Данные утекли / агент сделал не то

**Уровень: CRITICAL**

Продукт собирал и хранил персональные данные (телефоны, имена, названия компаний) из Telegram-каналов. SQLite-файл с этими данными лежал локально у каждого пользователя. Один пользователь залил базу в публичный GitHub, другой потерял ноутбук — и 3000 контактов швейных производств оказались в открытом доступе.

**Цена:** Жалобы владельцев данных → Роскомнадзор → штраф. Репутационный удар в нише, где всё держится на доверии и сарафанном радио.

**Что надо было сделать:** Хэшировать/маскировать персональные данные в базе. Не хранить телефоны в открытом виде. Автоматическое удаление данных старше N дней. Шифрование .db файла.

## O2. One-person bottleneck: ты заболел — продукт умер

**Уровень: HIGH**

Bus factor = 1. Один человек писал код, поддерживал зависимости, отвечал пользователям, чинил API. Когда ты заболел/выгорел/переключился — некому было обновить Telethon, починить сломанный парсинг, ответить на вопрос пользователя.

**Цена:** 3 недели простоя → 80% churn пользователей. Они нашли альтернативу или вернулись к ручному поиску.

**Что надо было сделать:** Open-source core + документация, чтобы сообщество могло подхватить. Или хотя бы: полный runbook для фрилансера, CI/CD с автотестами, чтобы любой мог запустить и проверить. Или признать: one-person без backup — это хобби.

## O3. Качество данных drift'ило без контроля

**Уровень: HIGH**

Со временем каналы меняли формат постов, появились новые типы объявлений, старые теги стали нерелевантными. Tagging schema drift'ила. Модель, которая правильно классифицировала «шьём джинсы» в январе, к июню путала «шьём» и «продаём».

**Цена:** К месяцу 6 точность классификации упала с 70% до 45%. Пользователи не заметили сразу — но перестали доверять результатам и ушли молча.

**Что надо было сделать:** Версионирование tagging schema. Лог ошибок классификации — еженедельный просмотр, что модель путает. Механизм ручной корректировки тегов пользователем с feedback loop. Периодический пересмотр правил экстракции.

## O4. Онбординг: 7 шагов до первого результата

**Уровень: MEDIUM**

Установить Python → установить CLI → получить API-ключи Telegram → настроить конфиг → добавить каналы → запустить парсинг → подождать. 7 шагов для не-технического пользователя. Каждый шаг — точка отсева.

**Цена:** Из 50 человек, которые заинтересовались, 35 бросили на шаге 2-3 (Python/API-ключи). 10 дошли до конца, 5 остались.

**Что надо было сделать:** Один бинарник (pyinstaller). Интерактивная `init` команда с пошаговыми инструкциями. Или: Telegram-бот, где онбординг = нажал /start. Каждый лишний шаг в онбординге = -30% конверсии.

## O5. Поддержка Telegram API — ongoing overhead

**Уровень: MEDIUM**

Telegram обновляет API, Telethon обновляется, зависимости ломаются. Для one-person проекта это 4-8 часов в месяц на поддержку, которые не генерируют revenue.

**Цена:** 6 часов/мес × 12 мес = 72 часа поддержки за $0 revenue. Это 72 часа, которые можно было потратить на поиск paying users.

**Что надо было сделать:** Автоматизированные тесты на парсинг — если API сломался, узнать сразу. Pin версий, обновлять только когда необходимо. Зарезервировать 20% времени на поддержку, 80% — на growth.
