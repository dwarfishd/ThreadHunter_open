## Вердикт

### 3 блокирующих риска (kill the project)

**1. CRITICAL — Telegram API: единственная точка отказа.**
Весь продукт на арендованной земле. Telethon — серая зона ToS. Бан сессии = смерть бизнеса. Это не митигируется, если вся архитектура завязана на один источник. Без мульти-источника проект нежизнеспособен.

**2. CRITICAL — Отсутствие Buyer Pain.**
Менеджеры швейных производств не чувствуют боли — они находят подрядчиков через WhatsApp и сарафанное. Парсер решает проблему, которой нет. Без валидации спроса (предоплата ДО разработки) это продукт для себя, не для рынка.

**3. CRITICAL — LLM-структурирование бесплатно встроено в ChatGPT/Claude.**
Ценностная пропорция продукта схлопнулась. То, что было «killer feature» — стало нативной функцией платформы за $0. Без уникальной базы данных или CRM/workflow поверх парсинга — не за что платить.

### 2 решаемых риска (fixable, но бессмысленно без решения блокирующих)

**4. HIGH — Legal/Compliance (персональные данные).**
Решаемо: юрист до запуска, маскирование данных, удаление старше N дней, парсинг только рекламных постов. Но это добавляет cost и friction к продукту, который и так не нужен рынку.

**5. HIGH — CLI барьер для не-технических пользователей.**
Решаемо: Telegram-бот вместо CLI, один бинарник, интерактивный онбординг. Но это ещё 2-4 недели разработки для продукта без validated demand.

---

## Итог

**Проект заблокирован тремя критическими рисками, каждый из которых смертелен сам по себе.** Даже если решить один — два других убьют. Даже если решить все три — TAM ($300-600 MRR) не оправдывает invested time.

**Рекомендация:** Убить. Вернуться к идее только после: (а) 15 оплаченных интервью с целевой аудиторией, (б) подтверждения, что LLM не commoditize весь pipeline, (в) мульти-источник данных, не зависящий от одной платформы.
