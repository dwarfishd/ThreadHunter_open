# Technical Risks — Post-Mortem

## T1. Telegram API: единственная точка отказа

**Уровень: CRITICAL**

Весь продукт построен на одной платформе. Telethon работает через MTProto — это серая зона ToS. Когда Telegram решил зачистить массовый scraping (бан сессий, rate limits, ограничение доступа к публичным каналам), продукт умер за один день.

**Цена:** $0 revenue после блокировки. Все месяцы разработки — в мусорку.

**Что надо было сделать:** Проектировать мульти-источник с первого дня: Telegram + веб-парсинг досок объявлений + ручная база данных. Fallback-план, а не единственная интеграция.

## T2. Качество AI-классификации на русском/кыргызском

**Уровень: HIGH**

Локальная LLM (Ollama/llama.cpp) и spaCy NER на русском — ~70% точности на чистых текстах. Тексты Telegram-каналов: сленг, сокращения, эмодзи, фото вместо текста, номера в формате «700 12-34-56». Результат: ложные теги, нераспознанные контакты, пользователи не доверяют данным.

**Цена:** 30-40% постов классифицированы неверно → пользователи перестали пользоваться через 2 недели.

**Что надо было сделать:** Начать с rule-based экстракции: регулярки для телефонов, гео-справочник для городов, keyword-матчинг для ассортимента. Это даёт 90% точности на 80% кейсов. LLM — только на edge cases, и только после валидации спроса.

## T3. Proactive парсинг сжигал деньги без монетизации

**Уровень: HIGH**

Архитектура «scheduled trigger → fetch → AI classify → store» означает, что каждый запуск стоит денег: LLM-запросы, серверное время, хранение SQLite. При 0 paying users это proactive bankruptcy.

**Цена:** $200-500/мес на инфраструктуру и токены × 6 месяцев = $1,200-3,000 сожжено без единого клиента.

**Что надо было сделать:** On-demand парсинг: пользователь запускает → получает результат. Никакого фонового сжигания до подтверждения PMF. Local First = local cost, но всё равно cost.

## T4. Кроссплатформенность CLI — больше времени, чем ожидал

**Уровень: MEDIUM**

Три ОС, три планировщика, разные нюансы Telethon. Вместо 2 недель на CLI — 6 недель на отладку `launchd` vs `cron` vs `Task Scheduler`.

**Цена:** 4 недели потерянного времени, которое можно было потратить на разговоры с клиентами.

**Что надо было сделать:** Один бинарник (pyinstaller), один сценарий (daily cron), одна ОС для MVP. Остальное — после первых paying users.

## T5. SQLite: один файл — один backup — одна потеря

**Уровень: MEDIUM**

Один файл .db = одна точка отказа. Пользователи не делали бэкапы. Один повреждённый файл — и вся база контактов за 3 месяца исчезла.

**Цена:** Потеря доверия одного ключевого пользователя → сарафанное радио в нише → минус 15 потенциальных клиентов.

**Что надо было сделать:** Автоматический бэкап при каждом запуске + CLI-команда `backup` с первого дня. Или хотя бы warning при init: «Вы отвечаете за бэкап».
