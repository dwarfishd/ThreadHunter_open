# Tech Decisions

## Принятые технические решения

### 1. Python + Telethon для Telegram API
**Решение:** Telethon (Python) — зрелая асинхронная библиотека, MTProto протокол.

**Альтернативы:**
- Pyrogram — тоже MTProto, но менее популярная, хуже документация
- GramJS (Node.js) — хорошая, но JavaScript экосистема слабее для NLP задач

**Почему:** Python даёт лучшую экосистему для NLP/классификации (spaCy, transformers, ollama). Telethon — самая зрелая библиотека с активным сообществом.

### 2. Локальная LLM для классификации
**Решение:** Ollama / llama.cpp — локальный инференс, данные не покидают машину.

**Альтернативы:**
- OpenAI API — быстрее и точнее, но нарушает Local First
- Rule-based (regex + keywords) — дешевле, но качество классификации значительно ниже

**Почему:** Соответствует принципу Local First. Пользователь контролирует данные. Современные локальные модели (7B-13B) достаточно хороши для NER и классификации тегов.

### 3. SQLite для хранения
**Решение:** Один файл SQLite, встраиваемая БД.

**Альтернативы:**
- PostgreSQL — требует сервер, нарушает «один человек управляет всем»
- JSON файлы — проще, но нет индексов, транзакций, запросов

**Почему:** SQLite — ноль конфигурации, работает везде, один файл для бэкапа. Достаточно для объёмов MVP (тысячи постов, не миллионы).

### 4. Alembic для миграций
**Решение:** Alembic — стандарт де-факто для Python, версионирование схемы, rollback.

**Альтернативы:**
- Ручные SQL-скрипты — проще, но нет автоматического rollback

**Почему:** При обновлении продукта пользователь просто запускает `alembic upgrade head`. Не нужно вручную применять SQL.

### 5. Typer для CLI
**Решение:** Typer — современный CLI фреймворк с type hints, автодокументацией.

**Альтернативы:**
- argparse — встроен, но verbose и старый
- Click — популярен, но Typer проще с type hints

**Почему:** Typer даёт лучший DX: type hints → авто-валидация → авто-help. Меньше бойлерплейта.

### 6. Системные планировщики (cron/launchd)
**Решение:** Использовать нативные OS-планировщики, не свой демон.

**Альтернативы:**
- Свой демон/сервис — сложнее, больше точек отказа
- APScheduler (Python) — работает только пока процесс жив

**Почему:** cron/launchd/Task Scheduler — надёжные, проверенные, переживают перезагрузку. Не изобретать велосипед.
